# CLAUDE.md — TelegramMediaRelayBot

## Что это
Telegram-бот: пользователь шлёт ссылку → бот скачивает медиа (yt-dlp / gallery-dl) → раздаёт контактам пользователя по его правилам. C# / .NET, Telegram.Bot, SQLite (Dapper + FluentMigrator), Docker. Лицензия AGPL-3.0.

Проект начинался как **TikTokMediaRelayBot** — первое имя и origin-story ценны автору. При любых переработках README/доков сохранять упоминание истории (раздел Origin и/или пасхалка). Не вычищать.

## Курс проекта (rework 2026)
1. **Сохранить ценность и историю** — origin story и первое название остаются в README/пасхалке.
2. **Минимализм** — ядро + базовые фичи: relay-флоу, контакты и инвайт-линки, access control («кто может пользоваться ботом»), авто-действия (default actions). Сложные generic-конфиги и спекулятивные абстракции — снос и переработка в простые формы.
3. **Качественная работа** — бот отвечает надёжно; Docker + локальный Bot API server для файлов до 2 ГБ; корректная работа yt-dlp (видео всегда со звуком) и gallery-dl.

## Архитектурные правила (обязательны для всего нового и переработанного кода)

**Слои (направление зависимостей строго внутрь):**
- `Domain` — бизнес-логика и модели. Ноль внешних зависимостей (ни Telegram.Bot, ни Dapper, ни yt-dlp).
- `Application` — use cases, порты (интерфейсы) к внешнему миру.
- `Infrastructure` — адаптеры: Telegram, yt-dlp/gallery-dl, SQLite, proxy/Tor. Реализуют порты.
- Composition root (Program.cs) — единственное место, где слои связываются.

**Dependency Inversion:** бизнес-логика зависит только от интерфейсов. Реализации подключаются в composition root. Запрещено: `new` инфраструктурных классов внутри логики, service locator (`GetRequiredService` вне composition root).

**SSOT:** каждое знание живёт ровно в одном месте.
- Конфиг — одна типизированная модель (Options pattern) с валидацией на старте. Никаких статических полей-конфигов, читаемых из любого места кода.
- Callback-имена, лимиты, задержки, пути — централизованные константы, не строковые литералы по коду.

**Ноль хардкода:** магические числа/строки → именованные константы или конфиг. Если значение может захотеться поменять без перекомпиляции — оно в конфиге.

**SOLID:**
- Один класс — одна ответственность. God-классы (текущий `TGBot`) при касании — разбивать.
- Интерфейсы по потребителю. Без крайностей: ни God-интерфейсов, ни 4 микро-интерфейсов на один агрегат без реальной нужды.

**Адаптеры для внешнего:** yt-dlp, gallery-dl, Telegram API, Tor/proxy — только за портами. Замена инструмента не трогает бизнес-логику. Типы Telegram.Bot (`Update`, `Message`, …) не проникают в Domain/Application — на границе конвертировать в свои DTO.

**Состояние:** без глобальных мутабельных статиков (текущие `Config.*`, `TGBot.cancellationToken`, статические менеджеры сессий — антипримеры, при касании переводить в DI-сервисы). Всё потокобезопасно.

**Ошибки:** `Result<T>` для ожидаемых ошибок, исключения для багов. Пустые catch запрещены. Всё логируется через Serilog.

**YAGNI:** никаких абстракций «на будущее». Антипример из прошлого — MySQL-заглушка без единой реализации. Фича добавляется, когда нужна, не раньше.

## Статус rework (июль 2026, ветка rework/cleanup)
Решено: «бот не отвечает» (poison-update без try/catch зацикливал polling на одном offset — теперь `UpdateHandler` ловит и пропускает; плюс supervised BackgroundService), no-audio (явный format в `YtDlpDownloader`), byte[]→файлы (стрим/`file://`, 2 ГБ через локальный Bot API), CTS-утечки, N+1 (JOIN), мёртвый код (RateLimited, MySQL-заглушка). Конфиг → типизированный AppConfig с валидацией. Загрузчики → `IMediaDownloader`-цепочка + per-host прокси. Персистентная очередь загрузок. Docker-стек с bot-api-форком (прокси до DC).

Осталось (отложено осознанно): декомпозиция god-класса `TGBot`, стейты на immutable (гонки латентны — апдейты последовательны), убрать статический мост `Config` в DI. README-драфт (.NET 8→10, origin-story). `.sln` не затрекан.
Известное ограничение: персистентная очередь = at-least-once (краш ровно между отправкой и удалением job → возможен дубль).
